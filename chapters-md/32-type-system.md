# Chapter 32: Type System

The type system defines the core data structures that flow through the runtime.

## Usage

The usage structure tracks token consumption for a single API call. It records:

- **Input tokens**: tokens sent to the model
- **Output tokens**: tokens generated by the model
- **Cache creation input tokens**: tokens written to the prompt cache
- **Cache read input tokens**: tokens served from the prompt cache
- **Server tool use**: optional metadata about server-side tool execution
- **Service tier**: optional identifier for the service tier used

## Tool Result

A tool result is the standard return value from any tool execution. It contains:

- A **type** field, always "tool_result"
- A **tool use identifier** linking the result to the tool call that produced it
- A **content** string with the output
- An **error flag** indicating whether the result represents a failure

## Tool Use Context

The tool use context is the shared state object passed to every tool invocation. It contains the session's options and configuration, providing tools with access to the runtime environment without requiring them to maintain their own state.

## Messages

The message types define the structure of conversation messages:

- **Assistant messages** represent the model's responses. They carry the message content (text and tool use blocks), duration in milliseconds, a unique identifier, an optional stop reason, and a flag indicating whether the message represents an API error.
- **User messages** represent the user's input or tool results. They carry content blocks and optional metadata about tool results.

Factory functions create these message types with appropriate defaults and identifiers. A special "user interruption message" type is used to signal errors back to the model when a tool call cannot be completed, carrying the tool use identifier and error message.
